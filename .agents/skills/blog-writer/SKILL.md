# Tech Blog Writer（日本語技術ブログ記事）

日本語技術ブログ記事を MDX で生成するスキル。

---

## ヒアリング（記事生成前に確認）

ユーザーから以下を確認する（会話から読み取れる場合は省略可）：

1. **トピック** — 何について書くか
2. **記事の種類** — 下記「記事タイプ別構成」を参照
3. **読者層** — 初心者向け／中級者向け／上級者向け
4. **コード例の有無** — 言語・フレームワーク
5. **参考資料** — URLや書籍名があれば

---

## フロントマター

```yaml
---
title: '記事タイトル'
date: 'YYYY-MM-DD'
excerpt: '記事の一言要約（1〜2文、検索スニペットになる）'
tags: ['タグ1', 'タグ2', 'タグ3']
---
```

**タイトルの推奨パターン（記事タイプ別）：**
| タイプ | タイトル例 |
|--------|-----------|
| 解説系 | `〇〇とは？仕組みと使い方を理解する` |
| 調査系 | `〇〇を調べてみた` / `〇〇がわからなかったので調べた` |
| 実装系 | `〇〇を使って△△を実装してみた` |
| 比較系 | `〇〇 vs △△ — 違いと使い分けを整理する` |
| アンチパターン系 | `〇〇のアンチパターンとその解決策` |

> [!NOTE]
> これらはあくまで推奨パターンであり、内容に合わせて適宜調整すること。

---

## 文体・トーン

- **一人称**：「私は」を多用せず、地の文で語る（`〜した`、`〜がある`、`〜と気づいた`）
- **文末**：です・ます調で統一。技術的説明は「〜である」も可
- **読者への語りかけ**：導入部のみ使用（`〜と感じたことはないでしょうか？`）。本文中は乱用しない
- **主観の出し方**：`〜と思う` より `〜ではないだろうか` / `〜と考えられる`
- **テンポ**：1段落3〜5文。長い説明は箇条書きに切る

## 本文構成

### 1. リード文（h1直下、本文冒頭）

```
〇〇を実装しているとき、△△という問題にぶつかった。

調べてみると〜ということがわかった。この記事では〜について解説する。
```

- **問題・動機を最初に示す**（なぜこれを書いたか）
- 記事で得られることを1〜2文で予告
- 長すぎないこと（200字以内が理想）

### 2. 本文（h2セクション）

#### 見出しのルール

- 基本的に**番号なしのシンプルな見出し**を推奨する
- セクション間の関係を明確にしたい場合のみ `## 1. 〇〇` のような番号付きを使用する
- h3は h2 の補足説明に使う（乱用しない）

#### セクション構成パターン

**解説系：**

```
## 1. 背景・問題
## 2. 〇〇とは
## 3. 基本的な使い方
## 4. 応用・実践例
## 5. 注意点・落とし穴
## まとめ
```

**アンチパターン系：**

```
## なぜこれが問題なのか
## アンチパターンの具体例
## 正しいアプローチ
## まとめ
```

**実装系：**

```
## 前提条件・環境
## 実装手順
### ステップ1
### ステップ2
## 動作確認
## まとめ・次のステップ
```

### 3. まとめ（必須）

```
## まとめ

- 要点1
- 要点2
- 要点3

〇〇については〜という結論になった。〜の場合は参考にしてみてほしい。
```

- 箇条書きで記事の要点を3〜5つ列挙
- 締めの1文で「誰の役に立つか」を示す

### 4. 参考資料（任意、あれば必ず付ける）

```
## 参考

- [タイトル](URL)
- [タイトル](URL)
```

## 特殊要素

### コードブロック

言語を必ず指定（`tsx`、`ts`、`bash`、`python` など）。重要な行にはコメントを添える。長いコードは分割し、各ブロックに説明文を付ける。

### 注意・補足ブロック

GitHub Flavored Markdown のアラート構文を使用：

```markdown
> [!NOTE]
> 補足説明や豆知識

> [!TIP]
> 実践的なヒント

> [!WARNING]
> 注意が必要な挙動や落とし穴

> [!CAUTION]
> 重大なミスにつながるリスク
```

### 比較テーブル

表の後に**一言の使い分けポイント**を必ず添える。

### 強調ルール

- `**太字**`：重要な用語・結論の核心・判断基準
- `` `インラインコード` ``：コード・ファイル名・コマンド・プロパティ名
- _斜体_：英語の正式名称（初出時）— 多用しない
- 下線・打ち消し線：使わない

## MDX に関する注意（コンパイルエラーの回避）

MDX は JSX を含むことができるため、プレーンテキスト中に HTML タグのような記述があるとコンパイルエラーが発生することがあります。

### HTML タグの記述

`<a>` や `<button>` などのタグを説明文で使う場合は、必ず **インラインコード（バックティックス）** で囲ってください。

```markdown
❌ <a> 要素はナビゲーションに使います。
✅ `<a>` 要素はナビゲーションに使います。
```

### リンクテキスト内での注意

特に、Markdown のリンクテキスト `[リンクテキスト](URL)` 内に `<tag>` のような記述を含めると、`expected image or link on stack` というエラーが発生する原因になります。

```markdown
❌ [MDN: <button>要素](URL)
✅ [MDN: `<button>`要素](URL) （または単にタグを削除する）
```

## 品質チェックリスト

記事生成後に以下を確認：

- [ ] リード文に「なぜ書いたか」が含まれているか
- [ ] 見出しだけ読んで記事の流れが把握できるか
- [ ] コードブロックに言語指定があるか
- [ ] まとめが箇条書き＋締め文になっているか
- [ ] 専門用語を初出時に説明しているか
- [ ] 1セクションが長すぎないか（目安: 400字以内/h2）
- [ ] HTMLタグ（`<a>`, `<button>` 等）がバックティックスで囲われているか

---

## 生成手順

1. ユーザーのトピック・記事タイプ・読者層を確認
2. 上記フォーマットに沿って MDX 記事を生成
3. `/src/content/posts/YYYY-MM-DD-slug.mdx` に保存
4. `present_files` ツールでファイルを提示

## ファイル名規則

`YYYY-MM-DD-topic-in-kebab-case.mdx`

例：`2026-04-22-react-19-actions-guide.mdx`
